## Executive Summary

The branch implements a well-designed TF scoring fix (switching from FSI Value to secrecy score) plus several additional improvements: V-Dem trend estimation, institutional gatekeeping restructuring, new indicators, country picker search, and XSS hardening. Three critical bugs were found in the frontend: the country picker fails for ~21 countries with leading-zero numeric IDs, the picker button displays rank metadata instead of clean names, and a metadata key mismatch silently ignores configured domain weights.

## Critical Issues

### [C1] Picker-to-map selection fails for ~21 countries with leading-zero numeric IDs

- **File:** `js/app.js:270,330,743`
- **Bug:** When selecting a country from the picker dropdown, the reverse lookup at line 743 returns the NUMERIC_MAP key as a string (e.g., `'004'` for Afghanistan). The map highlight filter at line 270 does `String(d.id) === String(numericId)`, but TopoJSON stores `d.id` as a number (e.g., `4`), so `'4' !== '004'`. Because `numericId != null`, the alpha3 fallback is not reached.
- **Impact:** ~21 countries (all with ISO numeric codes < 100, including AFG, ALB, DZA, AGO, ARG, AUS, AUT, BGD, BEL, BOL, BRA) cannot be highlighted or zoomed to on the map when selected from the picker. The panel incorrectly shows "This territory is too small to display on the world map."
- **Suggested fix:** Normalize the comparison: `Number(d.id) === Number(numericId)` at lines 270 and 330.
- **Confidence:** High
- **Found by:** Logic & Correctness (Frontend)

### [C2] Picker button text includes rank/score metadata after dropdown selection

- **File:** `js/app.js:745`
- **Bug:** `selectCountry()` at line 308 sets `pickerBtn.textContent = cd.name` (clean name like "France"). Then line 745 immediately overwrites it with `button.textContent = item.textContent`, which contains `"France (#3)"` (alpha sort) or `"3. France (42)"` (score sort). Map clicks do not trigger this overwrite, so the behavior is inconsistent between selection methods.
- **Impact:** The picker button displays formatting artifacts (rank numbers, scores) that are meant for the dropdown list only.
- **Suggested fix:** Remove line 745 and let `selectCountry`'s line 308 be the sole source of truth for button text.
- **Confidence:** High
- **Found by:** Concurrency & State

### [C3] `metadata.default_weights` key mismatch with `DOMAIN_KEYS`

- **File:** `js/app.js:98` cross-ref `data/scores.json:12`, `js/lib.js:6`
- **Bug:** `scores.json` metadata has key `resource_labor_extraction` but `DOMAIN_KEYS` in `lib.js` uses `resource_capture`. At line 98, `scores.metadata.default_weights['resource_capture']` returns `undefined`, falling back to `1 / DOMAIN_KEYS.length` (0.143).
- **Impact:** Currently masked because all default weights happen to be 0.143 (equal). Any future non-equal weight configuration would silently ignore the resource_capture weight.
- **Suggested fix:** Rename the key in `scores.json` metadata from `resource_labor_extraction` to `resource_capture`, or update the generation code in `score_countries.py`.
- **Confidence:** High
- **Found by:** Logic & Correctness (Frontend)

## Important Issues

### [I1] Inconsistent XSS escaping in innerHTML templates

- **File:** `js/app.js:494-507`
- **Bug:** `esc()` is applied to `related_jurisdictions_note` (line 506) but NOT to `d.justification_detail` (line 507), `ind.question`, `ind.label`, `ind.facts` (lines 494-495), country names in tooltip (line 181), or `d.justification` (line 502). All are inserted via innerHTML.
- **Impact:** Defense-in-depth gap. All data originates from self-generated `scores.json`, so not currently exploitable. However, if data sources ever contain HTML characters (e.g., corrupted CSV, compromised CDN TopoJSON), unsanitized HTML would render in the DOM.
- **Suggested fix:** Apply `esc()` consistently to all data-derived strings in innerHTML templates.
- **Confidence:** High
- **Found by:** Logic & Correctness (Frontend), Security, Error Handling, Contract & Integration

### [I2] Picker button text not reset when selecting a no-data country from the map

- **File:** `js/app.js:294-308`
- **Bug:** When `selectCountry()` is called for a country with no data (e.g., Antarctica), `selectedCountryCode` is set to `null` (line 295) and the function returns at line 299 without reaching the picker button update at line 308. The button continues displaying the previously selected country's name.
- **Impact:** UI state inconsistency: panel says "No data for Antarctica", `selectedCountryCode` is null, but picker button says "France".
- **Suggested fix:** Reset picker button text in the `!cd` branch.
- **Confidence:** High
- **Found by:** Concurrency & State

### [I3] `_load_fsi_csv()` NaN timestamp selects wrong methodology

- **File:** `scripts/score_countries.py:1177`
- **Bug:** `df.sort_values("scoring_timestamp")` places NaN values last by default. If any row has a NaN timestamp, `iloc[-1]` selects that row's `methodology_id` instead of the most recent actual timestamp.
- **Impact:** Wrong methodology edition selected for all FSI data, producing incorrect secrecy scores. Currently no NaN timestamps in production data.
- **Suggested fix:** `df.dropna(subset=["scoring_timestamp"]).sort_values(...)` before `iloc[-1]`.
- **Confidence:** Medium
- **Found by:** Logic & Correctness (Backend)

### [I4] `_load_fsi_csv()` reads CSV twice per pipeline run

- **File:** `scripts/score_countries.py:1188,1215`
- **Bug:** `load_fsi_data()` calls `_load_fsi_csv()` at startup, and `load_fsi_related_jurisdictions()` calls it again when processing GBR inside the country loop. Commit f970dad claims to "eliminate double V-Dem CSV read" but introduced a double FSI CSV read.
- **Impact:** Minor performance issue and inconsistency with stated refactoring intent.
- **Suggested fix:** Cache the DataFrame or pre-load related jurisdictions before the country loop.
- **Confidence:** High
- **Found by:** Logic & Correctness (Backend), Error Handling, Contract & Integration, Concurrency & State

### [I5] Schema declares `secrecy_score` as number but code can produce null

- **File:** `data/schema.json:125`, `scripts/score_countries.py:1197`
- **Bug:** `load_fsi_related_jurisdictions()` sets `secrecy_score` to `None` when `index_score` is NaN. Schema declares `"type": "number"` without allowing null.
- **Impact:** Latent schema violation. No nulls in current production data.
- **Suggested fix:** Change schema to `"type": ["number", "null"]` or filter out entries with null secrecy_score.
- **Confidence:** Medium
- **Found by:** Contract & Integration

### [I6] FSI fallback uses secrecy as value, producing misleading context fact

- **File:** `scripts/score_countries.py:1228-1230`
- **Bug:** When `index_value` is missing, `value` is set equal to `secrecy`. Downstream at line 1899, this displays as `"FSI Value (secrecy x scale): 63"` — labeling a secrecy score (0-100) as an FSI Value (which can be >1000).
- **Impact:** Misleading display for analysts who understand FSI methodology. Affects countries where FSI CSV has no `index_value` column.
- **Suggested fix:** Set `value` to `None` in the fallback, letting the downstream null-check skip the misleading context fact.
- **Confidence:** Medium
- **Found by:** Error Handling

### [I7] Duplicate NUMERIC_MAP entries: 356 and 699 both map to 'IND'

- **File:** `js/lib.js:69,123`
- **Bug:** Both entries map to India. Code 699 is non-standard (possibly Jammu & Kashmir in some TopoJSON datasets). The reverse lookup at line 743 of app.js uses `Object.entries().find()` which returns whichever comes first in property order.
- **Impact:** Minor behavioral inconsistency if TopoJSON has feature with id=699 separately from id=356.
- **Confidence:** Medium
- **Found by:** Logic & Correctness (Frontend)
