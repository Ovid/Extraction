## Important Issues

### [I1] XKX (Kosovo) silently dropped by overly broad aggregate-code regex
- **File:** `scripts/score_countries.py:1332`
- **Bug:** The regex `^X[0-9A-Z]` in `load_indicator()` is intended to filter ILO aggregate codes (X01-X99, XA1, etc.) but also matches `XKX` — Kosovo's ISO alpha-3 code. Since `load_indicator()` is shared by ALL indicator sources, Kosovo is silently dropped from every indicator dataset (World Bank and ILO alike).
- **Impact:** Kosovo cannot be auto-scored for any indicator-based domain. World Bank raw data files (wb_gini.csv, wb_wgi_reg_quality.csv, etc.) contain XKX records that are now filtered out. Note: Kosovo had no scores in the prior state either, so this is not a regression on this branch — but the regex makes it impossible to ever add Kosovo scoring without fixing it.
- **Suggested fix:** Narrow the regex to `^X[0-9]` (covers X01-X99) or enumerate ILO aggregate codes explicitly in `EXCLUDE_CODES`.
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration

### [I2] fetch() returns non-existent file path when API returns empty data
- **File:** `scripts/fetchers/ilo.py:88-91`
- **Bug:** When the ILO API returns zero records, the CSV is not written (lines 72-75), but lines 88-91 unconditionally include `output_path` in the returned `files` list. The caller (`fetch_all.py`) records this phantom path in `manifest.json` as a successfully fetched file.
- **Impact:** Manifest tracks a file that does not exist on disk. While the scorer handles missing files gracefully (returns empty DataFrame), the manifest becomes misleading provenance documentation.
- **Suggested fix:** Build `files = [str(meta_path.relative_to(raw_data_dir))]`, only prepend the CSV path when `not df.empty`.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases, Concurrency & State

## Suggestions

- **Stale CSV on empty re-fetch** (`ilo.py:71`): If a previous run wrote `ilo_labor_share.csv` and a subsequent fetch returns empty, the old CSV persists and the scorer silently uses outdated data. Consider deleting the existing CSV when `df.empty`. (Found by: Error Handling, Concurrency & State)
- **Test gap for empty-response return value** (`test_ilo_fetcher.py:53`): `test_empty_response_does_not_write_csv` does not capture or assert on the return value of `fetch()`, masking I2. (Found by: Error Handling)
- **Hardcoded endPeriod=2025** (`ilo.py:19`): The API URL will miss 2026+ data. The World Bank fetcher also hardcodes its range (`2010:2024`), so this is a project-wide pattern, but worth noting. (Found by: Logic & Correctness)
- **ESH in EXCLUDE_CODES with misleading comment** (`score_countries.py:887`): ESH (Western Sahara) is a valid ISO 3166-1 alpha-3 code but is labeled "ILO non-country code". The comment should clarify it's excluded as a disputed territory, or it could be removed from the exclusion list. (Found by: Contract & Integration)
- **Inconsistent dict access in parse_sdmx_csv** (`ilo.py:36`): `REF_AREA` and `TIME_PERIOD` use direct `row[]` access (raises KeyError if missing) while `OBS_VALUE` uses `.get()`. Low practical impact since the API schema is stable, but inconsistent. (Found by: Error Handling, Security)
