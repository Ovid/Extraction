## Important Issues

### [I1] ILO data falsely attributed to "World Bank data" in justification text
- **File:** `scripts/score_countries.py:1526`
- **Bug:** `build_technical_justification("World Bank data", ind_info)` hardcodes the source label. Since ILO indicators flow through `build_wb_domain()` alongside World Bank indicators, the justification for labour income share reads: `"Auto-scored from World Bank data. Gini Index: 41.800 (normalized: 46); Labour income share of GDP: 55.807 (normalized: 30)."` -- attributing ILO data to the World Bank.
- **Impact:** Incorrect provenance in user-facing data. Visible in every country's `economic_concentration.justification_detail` field.
- **Suggested fix:** Add a `"source_label"` field to each INDICATOR_CONFIG entry and derive the label from the group's actual sources, or pass the source label through to `build_wb_domain`.
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration

### [I2] "World Bank" always appended to source_names, ILO never mentioned
- **File:** `scripts/score_countries.py:1673`
- **Bug:** `source_names.append("World Bank")` runs whenever `country_data` is non-empty, regardless of which sources are actually present. The notes field for every country reads e.g. `"Auto-scored from RSF, TJN, V-Dem, World Bank (7/7 domains covered)."` -- ILO is never credited.
- **Impact:** ILO is invisible as a data source in the output. Countries with only ILO data (no WB data) would incorrectly credit "World Bank."
- **Suggested fix:** Detect which source providers are actually present in `country_data` (e.g., by checking source_key prefixes or adding a source_name field to INDICATOR_CONFIG) and build source_names from that.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I3] Countries with only ILO data display ISO code instead of name
- **File:** `scripts/score_countries.py:1657-1665` and `scripts/fetchers/ilo.py:66`
- **Bug:** ILO fetcher sets `country_name` to `""` for all rows. The name resolution at lines 1661-1663 correctly filters out empty strings, but if a country only appears in ILO data (no World Bank data) and is not in `COUNTRY_NAME_OVERRIDES`, the name falls through to the raw 3-letter ISO code (e.g., "PRK" instead of "North Korea").
- **Impact:** Some countries could display with their ISO code instead of human-readable name. Mitigated by `COUNTRY_NAME_OVERRIDES` covering many edge cases and most countries having World Bank data.
- **Suggested fix:** Verify that `COUNTRY_NAME_OVERRIDES` covers all ILO-only countries, or have the ILO fetcher resolve country names from a separate lookup.
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Error Handling & Edge Cases
